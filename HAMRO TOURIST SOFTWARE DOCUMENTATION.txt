1. Executive Summary & Vision
Hamro Tourist is a multi-tenant Software-as-a-Service (SaaS) platform purpose-built for travel agencies. It empowers agencies to manage their entire product catalogue — tours, treks, itineraries, activities, and packages — publish a fully branded website, and operate a CRM, all from a single dashboard. The platform is hosted on hamrotourist.com and each agency receives their own subdomain (agencyname.hamrotourist.com) or can connect a custom domain (agencyname.com) with full white-label admin access (admin.agencyname.com).

1.1 Core Value Proposition
•	Zero-infrastructure travel website in minutes (Shopify-equivalent for travel agencies)
•	Rich product modeling: Tours, Treks, Activities, Packages with complex pricing
•	Theme-based website builder with real-time preview and SEO controls
•	Subdomain-first, custom domain-ready architecture via Cloudflare
•	Plan-based feature gating (Free → Pro → Enterprise)
•	White-label admin panel per agency after domain connection

2. High-Level System Architecture
The architecture is a domain-driven, microservices-based system fronted by an API Gateway, using a multi-tenant PostgreSQL strategy (shared database, schema-per-tenant), Cloudflare for DNS/CDN/routing, and Next.js for both the admin SPA and the dynamic agency website renderer.

2.1 Architecture Philosophy: Why Microservices?
We chose a microservices architecture for the following strategic reasons:
•	Independent Scalability — The website renderer service will get high read traffic (visitors browsing agency sites), while the CRM and product services are write-heavy but low-frequency. Microservices let us scale each independently.
•	Team Isolation — Each service has a clear owner and bounded context. The DNS/Domain service can be worked on independently of the Booking engine.
•	Deployment Safety — A bug in the notification service cannot crash the website renderer. Services fail independently.
•	Technology Flexibility — The website renderer can use Next.js SSR/SSG while internal services use NestJS REST/GraphQL APIs.
•	Plan Enforcement — The Plan/Billing service is a separate gate that all other services query, making plan enforcement consistent and centralized.

2.3 Request Flow Diagrams
2.3.1 Agency Visitor Requests (agencyname.hamrotourist.com)
When a visitor lands on an agency website, the following flow occurs:
1.	Browser sends GET request to agencyname.hamrotourist.com
2.	Cloudflare DNS resolves the wildcard *.hamrotourist.com CNAME to the Website Renderer cluster
3.	Cloudflare Worker intercepts and reads the subdomain prefix (agencyname)
4.	Worker forwards to Next.js Renderer with X-Tenant-Slug: agencyname header
5.	Next.js middleware reads slug, queries Website Builder Service for theme config, layout, published status
6.	Product Service is queried for published tours, treks, packages
7.	Page is rendered server-side (SSR) or from ISR cache and returned to visitor

ISR (Incremental Static Regeneration) means the first visitor gets SSR; subsequent visitors get the cached static version until the agency re-publishes or the cache TTL expires (e.g., 5 minutes). This gives the performance of static + freshness of SSR.

2.3.2 Custom Domain Request (agencyname.com)
8.	Agency adds CNAME record: @ or www → proxy.hamrotourist.com
9.	Domain Service verifies ownership via DNS TXT record (_hamrotourist-verify.agencyname.com)
10.	After verification, Cloudflare API issues a Custom Hostname certificate (via Cloudflare for SaaS)
11.	Cloudflare stores the domain → tenant_slug mapping in its Custom Hostnames table
12.	All future requests to agencyname.com follow the same renderer flow with tenant resolved from the custom hostname lookup
13.	admin.agencyname.com is also CNAMEd → admin-proxy.hamrotourist.com and serves the Admin Frontend with the tenant context pre-loaded
 
3. Cloudflare Architecture — DNS, Domains & CDN
Cloudflare is the backbone of the entire routing and domain management strategy. Here is a detailed explanation of every Cloudflare component used and why.

3.1 Cloudflare for SaaS (Custom Hostnames)
This is the key Cloudflare product that powers custom domain support. It allows hamrotourist.com to act as a reverse proxy for thousands of agency custom domains — without managing individual DNS zones for each.

•	You add hamrotourist.com zone to Cloudflare (this is your zone).
•	You create a Fallback Origin: proxy.hamrotourist.com → your Website Renderer servers.
•	When an agency connects agencyname.com, your Domain Service calls the Cloudflare API to create a Custom Hostname entry for agencyname.com.
•	Cloudflare issues a free TLS certificate for agencyname.com and routes traffic through your fallback origin.
•	The agency only needs to add one CNAME record at their registrar: www → proxy.hamrotourist.com

3.2 DNS Record Architecture (hamrotourist.com zone)

CNAME	app.hamrotourist.com	Admin cluster	Shared admin panel

3.3 Cloudflare Workers — Tenant Resolution
A Cloudflare Worker runs at the edge (before requests hit your servers) and handles tenant resolution:

// cloudflare-worker/tenant-router.js
export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    const host = url.hostname;

    // Case 1: subdomain of hamrotourist.com
    if (host.endsWith('.hamrotourist.com')) {
      const slug = host.replace('.hamrotourist.com', '');
      if (slug === 'app' || slug === 'api') return fetch(request);
      return fetch(request, {
        headers: { ...request.headers, 'X-Tenant-Slug': slug }
      });
    }

    // Case 2: custom domain — lookup from KV store
    const tenantSlug = await env.DOMAIN_MAP.get(host);
    if (tenantSlug) {
      return fetch(request, {
        headers: { ...request.headers, 'X-Tenant-Slug': tenantSlug }
      });
    }
    return new Response('Agency not found', { status: 404 });
  }
}

Cloudflare KV (Key-Value store) is used as a lightweight domain→tenant mapping cache. When an agency connects a custom domain, the Domain Service writes to KV: { 'agencyname.com' → 'agencyname-slug' }. Workers read KV in ~1ms globally with zero cold starts.

3.4 Cloudflare R2 — Media Storage
Cloudflare R2 is an S3-compatible object storage with zero egress fees. All agency-uploaded images (banners, tour photos, activity images) are stored in R2 and served via a CDN URL. Each agency gets a folder prefix: /tenants/{agency-slug}/media/

4. Database Architecture — Multi-Tenancy Strategy
We use the Shared Database, Schema-per-Tenant pattern. This means: one PostgreSQL cluster, but each agency has an isolated PostgreSQL schema (e.g., schema name = tenant_acme_travels). This gives us strong data isolation without the overhead of per-tenant database clusters.

4.1 Why Schema-per-Tenant?
Strategy	Pros	Cons	Verdict
Shared DB + shared tables (row-level)	Cheapest, simplest migrations	Risk of data leakage; complex RLS policies	Not chosen — security risk
Schema-per-tenant (chosen)	Strong isolation; simple queries; easy backup per tenant; migrate single tenant	Migration must run per schema; slightly more setup	CHOSEN — best balance
Database-per-tenant	Maximum isolation; separate scaling	Very expensive at scale; complex ops	Future Enterprise tier only

4.4 Pricing Model — Deep Dive
The pricing system is fully custom and hierarchical. Here is how it works end-to-end for a trek:
•	A trek has multiple Itineraries (e.g., '8-Day Classic Route', '12-Day Extended Route').
•	Each Itinerary has multiple Pricing Tiers (e.g., 'Standard Season', 'Peak Season', 'Early Bird').
•	Each Pricing Tier has a base price and optional Group Discounts (e.g., 5–8 pax: 10% off; 9+ pax: 20% off).
•	Activities attached to the trek/tour can be 'included' (price absorbed) or 'optional' (customer chooses at checkout; activity.extra_price is added to cart).

Checkout price calculation: (Itinerary base price × pax) × group_discount_multiplier + SUM(optional_activities.extra_price per chosen activity × pax who selected it). This is computed server-side and never trusted from the client.

8. API Gateway & Service Communication

8.1 API Gateway Responsibilities
•	Authentication verification — validates JWT on every request before forwarding to services
•	Tenant Context Injection — extracts tenant slug from JWT claims or X-Tenant-Slug header; injects into all upstream requests
•	Rate Limiting — per tenant, per endpoint (free plan: 100 req/min; pro: 1000 req/min)
•	Request Routing — routes /products/* → Product Service, /auth/* → Auth Service, etc.
•	CORS — configured to allow requests from *.hamrotourist.com and tenant custom domains
•	Logging — every request logged with tenant_id, user_id, endpoint, latency for audit trail

8.2 Inter-Service Communication
Pattern	Use Case	Technology
Synchronous REST	CRUD operations, plan limit checks, config reads	Axios / @nestjs/axios with service discovery
Asynchronous Events	DNS verification complete, image processed, email send	BullMQ queues (Redis-backed)
Event Bus (future)	Booking created, payment confirmed, plan upgraded	Redis Pub/Sub or RabbitMQ
GraphQL (public)	Agency website reads product data (read-only, cached)	Apollo Server in Product Service

8.3 Tenant Context Pattern
Every API call carries tenant context. Here is the NestJS implementation pattern:
// tenant-context.interceptor.ts
@Injectable()
export class TenantContextInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler) {
    const request = context.switchToHttp().getRequest();
    const tenantSlug = request.headers['x-tenant-slug']
                    || request.user?.tenantSlug;
    if (!tenantSlug) throw new UnauthorizedException('No tenant context');
    // Set PostgreSQL search_path for this request's DB connection
    TenantContext.setSchema(`tenant_${tenantSlug}`);
    return next.handle();
  }
}

14. AI-Driven Development Roadmap
Because AI is building this platform, development is structured by feature complexity and dependency order — not calendar time. Each phase must be fully working and tested before the next begins. AI builds, validates, and iterates within each phase before progressing.

Phase 1 — Core Infrastructure & Data Layer
Everything else depends on this. AI builds the foundation first: no UI, no product features — just the skeleton that all other phases plug into.
•	Monorepo scaffold (Turborepo): all service folders, shared-types package, shared tsconfig
•	PostgreSQL setup: public (master) schema with tenants, users, plans tables
•	Auth Service: registration, login, JWT (RS256) with access + refresh tokens, RBAC roles
•	Tenant Service: company onboarding, schema provisioning (CREATE SCHEMA tenant_{slug}), plan limits API
•	Tenant migrations runner: CLI tool that applies tenant schema migrations to every existing tenant schema
•	API Gateway: routing, JWT validation middleware, tenant context injection, rate limiting scaffold
•	Redis setup: session store + BullMQ queue worker scaffold
•	Docker Compose: all services running locally with hot reload
•	GitHub Actions CI: lint, typecheck, unit tests on every PR

Phase 2 — Product Data Model & APIs
The full travel product domain — every entity, every relationship, every business rule — built as clean REST APIs with no frontend yet. AI validates all edge cases (plan limits, pricing overlaps, activity linking) at the API level first.
•	Product Service: Countries (max 5 free), Regions (max 3/country free) — full CRUD with plan enforcement
•	Tours and Treks: CRUD, linked to regions, status (draft/published), difficulty, cover image field
•	Activities: independent entity with base_price, type enum, bookable standalone flag
•	Activity linking: attach activity to tour/trek with inclusion_type (included | optional) and extra_price
•	Packages: cross-region product, many-to-many with regions via package_regions junction
•	Itinerary builder: parent polymorphic (tour | trek | package), multiple itineraries per product, day-by-day breakdown
•	Pricing engine: multiple tiers per itinerary (label, base_price, currency, valid_from/to)
•	Group discounts: non-overlapping pax ranges per pricing tier, discount_type (% | fixed)
•	Checkout price calculator: server-side function — (base × pax × group_multiplier) + optional activities
•	Full OpenAPI (Swagger) docs auto-generated for all endpoints

Phase 3 — Media, Storage & Cloudflare Routing
Connects the platform to Cloudflare's infrastructure. After this phase, agency subdomains work end-to-end.
•	Media Service: presigned R2 upload URLs, Sharp image variants (thumbnail/card/hero in WebP), CDN URL registry
•	Cloudflare R2 bucket setup with public CDN subdomain (media.hamrotourist.com)
•	Cloudflare Worker: wildcard *.hamrotourist.com tenant slug extraction + X-Tenant-Slug header injection
•	Cloudflare KV namespace: DOMAIN_MAP for custom domain → tenant slug lookups
•	Website Renderer (Next.js): tenant resolution from header, ISR with 5-min TTL, placeholder pages for all routes
•	Basic routing: /, /tours, /tours/[slug], /treks, /treks/[slug], /activities, /packages, /regions/[slug]
•	On-demand ISR revalidation endpoint: POST /api/revalidate?slug=X&secret=Y

Phase 4 — Website Builder & Theme System
The most complex frontend phase. AI builds the theme component libraries and the builder service in parallel, then wires them together.
•	Website Builder Service: website_configs table (theme, SEO, sections JSONB, colors, fonts, nav, footer)
•	Theme 1 — Adventure Bold: dark dramatic UI, fullscreen hero, bold typography
•	Theme 2 — Serene Journey: light minimal, photography-first, clean whitespace
•	Theme 3 — Heritage Classic: warm tones, traditional agency aesthetic, serif fonts
•	ThemeProvider in renderer: reads tenant theme config, renders correct component tree
•	All 10 section types implemented per theme: Hero, Featured Tours, Region Grid, Activity Cards, Packages, Testimonials, About, Why Us, Contact Form, Newsletter
•	Publish flow: validate config → write DB → clear ISR cache → set published=true → return live URL
•	SEO generation: OpenGraph tags, JSON-LD (TouristAttraction, TouristTrip), sitemap.xml, robots.txt per agency

Phase 5 — Admin Frontend (Dashboard)
The full agency-facing dashboard. AI builds module by module, each fully functional before moving to the next.
•	Onboarding wizard: sign up → company name/slug → theme pick → first country → first region → first product → publish
•	Geography Manager: country and region CRUD with plan limit warnings in UI
•	Product Manager: Tours, Treks, Activities, Packages — full CRUD forms with image upload
•	Itinerary Builder: multi-itinerary editor per product, drag-and-drop day ordering, pricing tier table
•	Group Discount Editor: pax range inputs with overlap validation, live price preview
•	Activity Linker: attach activities to tours/treks, toggle included/optional, set extra_price
•	Website Builder UI: section list with drag-and-drop reorder (dnd-kit), inline text editing, live iframe preview, theme switcher, color/font picker
•	Media Library: upload, view, delete, copy CDN URL — scoped per tenant
•	SEO Settings panel: title, description, OG image, favicon
•	Publish button with confirmation and live URL display

Phase 6 — Custom Domains, CRM & White-Label Admin
The growth features that enable agencies to go fully independent from the hamrotourist.com subdomain.
•	Domain Service: ownership verification via DNS TXT polling (BullMQ job, 60s interval, 24h timeout)
•	Cloudflare for SaaS integration: POST /custom_hostnames on verification success, TLS cert auto-issued
•	KV write on domain activation: DOMAIN_MAP.put(domain, slug)
•	admin.domain.com Custom Hostname: white-label admin panel with agency logo on login screen
•	Guided domain connection UI: step-by-step DNS instructions with real-time verification status polling
•	CRM Service: contacts, leads (Kanban pipeline), notes per contact, activity timeline, tags, search
•	Contact form → CRM lead auto-creation when visitor submits agency website contact form
•	Notification Service: onboarding sequence emails, domain-verified alert, contact form notification to agency

Phase 7 — Billing, Plan Enforcement & Analytics
Monetisation layer. Without this phase the platform cannot generate revenue.
•	Stripe integration: subscription plans (Free/Pro/Business), checkout session, webhook handlers for plan changes
•	Plan enforcement hardening: every service calls /tenant/:slug/limits before resource creation; returns 402 with upgrade CTA on limit hit
•	Billing portal: invoice history, plan upgrade/downgrade, payment method management (Stripe Customer Portal)
•	Analytics Service: page view tracking (privacy-first, no cookies), top tours/treks by views, CRM lead conversion rate
•	Agency analytics dashboard: traffic, top products, enquiry counts, plan usage meters

Phase 8 — Booking Engine & Payments 
Customer-facing booking. This phase is scoped for future build once Phases 1–7 are stable and generating revenue.
•	Availability calendar: date-blocking per itinerary, pax limits, open/closed status
•	Booking request flow: customer selects itinerary → chooses optional activities → enters group size → gets price → submits request
•	Payment gateway: Stripe (international) + eSewa / Khalti (Nepal market)
•	Booking confirmation emails to customer and agency
•	Agency booking management dashboard: view, confirm, cancel, refund bookings
•	Ticketing platform integration (events, one-day trips) — as announced in original scope
 
15. Key Architectural Decisions & Rationale

Decision	What We Chose	What We Rejected	Why
Tenancy Model	Schema-per-tenant	Row-level security	Schema isolation is safer; no accidental data leakage; easy per-tenant backup
Rendering Strategy	Next.js ISR	Pure SSR or pure SPA	ISR gives static performance (fast for visitors) with dynamic freshness (updates on publish)
Custom Domains	Cloudflare for SaaS	Nginx per-domain config	Cloudflare automates TLS cert issuance, routing, and global CDN for every custom domain at scale
Service Communication	REST + BullMQ (async)	gRPC or GraphQL federation only	REST is simpler to start; BullMQ handles unreliable async operations (DNS polling, emails) reliably
Database per Service?	Shared PostgreSQL, separate schemas	Separate DB per microservice	Multiple DBs adds ops complexity early. Schema isolation gives 80% of the isolation benefit at 20% of the cost
Frontend Framework	Next.js for both admin and renderer	Vue / Nuxt / Remix	One ecosystem to hire for; Next.js excels at both SSR (renderer) and SPA-like (admin)
Theme System	React component sets	CSS-only themes	Component-level themes allow section-level customization; CSS-only would be too rigid
Monorepo	Turborepo monorepo	Separate repos per service	Shared types (TypeScript interfaces) eliminate integration bugs; atomic commits across services
Storage	Cloudflare R2	AWS S3	Zero egress fees at Cloudflare CDN edge. S3 charges for every GB transferred; R2 does not
